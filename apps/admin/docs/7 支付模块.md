# 支付模块设计文档

本文档旨在详细阐述项目的支付功能实现，涵盖从发起支付到接收异步通知并完成业务处理的完整流程。

## 1. 核心流程概览

本支付系统采用微信 Native 支付（扫码支付）方式。核心流程如下：

1.  **发起支付**：前端携带商品信息请求后端的支付接口。
2.  **创建订单**：后端服务接收到请求，验证参数，创建系统内部的订单记录，并生成唯一的订单号。
3.  **获取支付凭证**：后端向微信支付服务器请求，生成一个用于支付的二维码链接。
4.  **返回二维码**：后端将二维码信息返回给前端，前端负责渲染该二维码供用户扫描。
5.  **用户支付**：用户使用微信扫描二维码，输入密码完成支付。
6.  **异步通知**：微信支付服务器在用户支付成功后，会向后端预设的通知地址（`notify_url`）发送一个 POST 请求，告知支付结果。
7.  **处理通知**：后端接收并验证该通知的合法性，如果支付成功，则更新内部订单状态，并执行与该订单关联的后续业务逻辑（例如：为用户开通会员）。
8.  **响应微信**：后端向微信支付服务器返回一个确认响应，告知已成功接收通知。若未返回或返回失败，微信会重试通知。

## 2. 文件职责说明

以下是构成支付模块的核心文件及其职责：

| 文件路径                               | 主要职责                                                                                                 |
| :------------------------------------- | :------------------------------------------------------------------------------------------------------- |
| `/core/controllers/pays_controller.ts` | **控制器层**：作为 HTTP 请求的入口，负责接收、验证前端请求，并调用服务层完成支付的发起和异步通知的处理。 |
| `/core/services/order_service.ts`      | **订单服务**：负责所有与订单相关的数据库操作，如创建订单、生成订单号、根据订单号查询订单等。             |
| `/core/services/pay_service.ts`        | **支付服务**：封装了与第三方支付（微信支付）交互的逻辑，并协调支付成功后的业务处理流程。                 |
| `/config/pay.ts`                       | **支付配置**：定义了不同类型的订单（`orderable_type`）与具体业务处理逻辑类之间的映射关系。               |
| `/core/pay/process/subscribe.ts`       | **具体业务处理**：实现了特定类型订单（如 `subscribe`）在支付成功后的具体业务逻辑和价格计算。             |

## 3. 详细流程解析

### 3.1. 发起支付

这是用户点击“购买”后触发的流程。

- **Endpoint**: `POST /api/pay` (根据路由和 `handle` 方法推断)
- **Controller**: `PaysController.handle`

**处理步骤**:

1.  **参数校验**: 控制器首先使用 `payValidator` 验证来自请求 `body` 的参数。
    - `subject` (string): 订单标题，例如 "商品-VIP会员"。
    - `orderable_type` (string): 可支付对象的类型，例如 "subscribe"。这个值对应 `config/pay.ts` 中的键。
    - `orderable_id` (number): 可支付对象的 ID，例如具体某个 VIP 套餐的 ID。

2.  **获取价格**: 调用 `payService.getPrice(orderable_type, orderable_id)`。
    - `PayService` 会根据 `orderable_type` 从 `config/pay.ts` 找到对应的处理类（例如 `SubscribeProcess`）。
    - 然后调用该类的 `getPrice(orderable_id)` 方法来获取精确的价格。

3.  **创建订单**: 调用 `orderService.create(...)`。
    - `OrderService` 内部会调用 `generateOrderSn()` 生成一个唯一的订单号（例如 `U1-1678886400000`）。
    - 将订单信息（包括用户ID、订单号、价格、关联对象等）存入数据库，初始支付状态为 `false`。

4.  **请求微信支付**: 调用 `payService.pay().transactions_native(params)`。
    - `PayService` 构造向微信支付服务器请求所需的参数，关键参数包括：
      - `description`: 订单描述 (来自 `subject`)。
      - `out_trade_no`: 商户订单号 (即 `order.sn`)。
      - `notify_url`: 支付结果异步通知地址。
      - `amount.total`: 支付金额，单位为分（`price * 100`）。
    - 发送请求后，微信返回包含 `code_url` 的结果。

5.  **生成二维码并返回**:
    - 使用 `qrcode.toDataURL(result.data.code_url)` 将微信返回的 `code_url` 转换成 Base64 格式的图片数据。
    - 最后，控制器向前端返回 `{ sn: order.sn, qrImg: 'data:image/png;base64,...' }`。

### 3.2. 接收并处理异步通知

用户支付成功后，微信服务器会调用我们的 `notify_url`。

- **Endpoint**: `POST /api/pay/notify` (根据路由和 `notify` 方法推断)
- **Controller**: `PaysController.notify`

**处理步骤**:

1.  **解密通知**: 微信发送的通知内容是加密的。控制器调用 `payService.pay().decipher_gcm(...)` 方法，使用商户密钥解密收到的 `resource` 数据，得到明文的支付结果 `params`。

2.  **获取订单**: 使用解密出的 `params.out_trade_no` (商户订单号) 调用 `orderService.getOrderBySn(...)` 从数据库中查询对应的订单。

3.  **校验和更新订单**:
    - 进行关键判断 `if (!order.payState && params.trade_state === 'SUCCESS')`，确保：
      - 订单未被处理过 (`!order.payState`)，防止重复执行业务逻辑。
      - 微信通知的交易状态为成功 (`'SUCCESS'`)。
    - 如果条件满足，则更新订单记录：
      - `order.payState = true`
      - `order.payType = 'wepay'`
      - `order.tradeNo = params.transaction_id` (微信支付订单号)
      - 调用 `order.save()` 保存更改。

4.  **执行后续业务**: 调用 `payService.processPay(order)`。
    - `PayService` 会根据订单的 `orderableType` 从 `config/pay.ts` 找到对应的处理类（例如 `SubscribeProcess`）。
    - 然后调用该类的 `handle(order)` 方法。**这是执行具体业务逻辑的地方**，例如在 `SubscribeProcess.handle` 中，可以编写为用户账户开通 VIP 权限的代码。

5.  **响应微信服务器**:
    - 整个 `try` 块执行成功后，向微信返回 `response.status(200).send({ code: 'SUCCESS', message: '成功' })`。
    - 如果在处理过程中发生任何错误（`catch` 块），则返回 `response.status(500).send({ code: 'FAIL', message: '失败' })`，并记录错误日志。

## 4. 如何扩展新的支付类型？

如果未来需要添加一种新的可支付商品（例如 "course"），只需遵循以下步骤：

1.  **创建处理类**: 在 `/core/pay/process/` 目录下创建一个新的文件，例如 `course.ts`。

    ```typescript
    // /core/pay/process/course.ts
    import Order from '#app/models/order'
    import Course from '#app/models/course' // 假设有课程模型

    export default class CourseProcess {
      // 支付成功后，为用户关联课程
      async handle(order: Order) {
        // 编写业务逻辑，例如：
        // await UserCourse.create({ userId: order.userId, courseId: order.orderableId });
        console.log(`为用户 ${order.userId} 开通课程 ${order.orderableId}`)
      }

      // 根据课程ID从数据库获取价格
      async getPrice(id: number): Promise<number> {
        const course = await Course.findOrFail(id)
        return course.price
      }
    }
    ```

2.  **注册处理类**: 在 `/config/pay.ts` 文件中，将新的处理类与一个键名关联起来。

    ```typescript
    // /config/pay.ts
    import SubscribeProcess from '#core/pay/process/subscribe'
    import CourseProcess from '#core/pay/process/course' // 引入新的处理类

    const payConfig = {
      process: {
        subscribe: SubscribeProcess,
        test: SubscribeProcess,
        course: CourseProcess, // 添加新的映射
      },
    }

    export default payConfig
    ```

3.  **前端调用**: 前端在发起支付时，`orderable_type` 传递 "course"，`orderable_id` 传递对应的课程 ID 即可。整个后端的处理流程会自动适配。
